# Recognition And Visibility Plan

## Purpose

This plan is meant to help the project become visible in a way that also protects attribution.

The main rule is simple: build a citable public record first, then expand visibility through targeted scientific contact.

## Core Goal

If V010 or a close equivalent later proves experimentally important, there should already be a clear public trail showing:

- who packaged the candidate and evidence
- when the public claim set was released
- what conditions were emphasized
- how the project moved from modeled evidence to external validation

## Recognition Stack

Recognition is strongest when three layers exist together:

1. authorship: the public repository and citation metadata identify Clayton Gray
2. timestamp: tagged public releases establish when the package was made public
3. archival citation: a DOI-backed archive makes the release easier to cite in formal contexts

No single layer is enough by itself.

## Visibility Principle

At this stage, the project should not chase broad attention first.

The better order is:

1. create a stable public record
2. make that record citable
3. put it in front of qualified labs, reviewers, and collaborators
4. publish milestone updates as the validation process advances

That approach produces better scientific credibility and better attribution discipline.

## First 30 Days

### Days 1 To 3

- publish the public repository
- confirm that `CITATION.cff` matches the final repo URL and authorship
- create the first tagged release for the public package
- write short release notes describing the evidence boundary and next validation step

### Days 4 To 7

- connect the repository to Zenodo or another DOI archiving service
- archive the first tagged release and record the DOI in the repository
- add a short citation section to the README once the DOI exists

### Days 8 To 14

- contact a small number of qualified labs or academic groups
- use the one-page external brief as the main technical summary
- ask for a narrow next step such as a synthesis quote, feasibility review, or lab referral

### Days 15 To 21

- document any serious feedback from labs or researchers
- tighten the public package if reviewers find unclear claims or missing evidence links
- keep all public edits consistent with the evidence boundary

### Days 22 To 30

- publish a short status update even if no lab commitment exists yet
- record whether the outcome was interest, referral, quote, or rejection
- preserve that update as part of the public project trail

## What To Publish Publicly

Public updates should emphasize:

- the current project status
- the exact next milestone
- whether external review has begun
- whether the project received a synthesis quote, technical feedback, or lab referral

Public updates should not imply experimental validation before it exists.

## Best Audience Order

The strongest early audience is:

1. qualified MOF or adsorption researchers
2. labs that can assess synthesis feasibility
3. technically credible collaborators who can inspect the evidence package
4. only after that, broader sponsors or program-level supporters

This order is better for both recognition and scientific seriousness.

## Recognition Safeguards

To improve recognition if the candidate later proves important:

- keep Clayton Gray listed in public citation metadata
- avoid fragmenting the public record across many disconnected documents
- link every outreach conversation back to the same public repository
- tag releases before major public outreach pushes
- archive meaningful milestones with release notes rather than silent edits

## What Success Looks Like

Near-term success does not require experimental proof.

At this stage, strong success would be:

- one citable public release
- one DOI-backed archive
- one qualified lab review or synthesis quote
- one documented public milestone update

That combination creates a credible attribution trail even before physical validation is complete.
