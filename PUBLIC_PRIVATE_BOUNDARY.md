# Public And Private Boundary

## Publish Publicly

These materials are generally appropriate for a public GitHub repository:

- workflow code and scripts that do not expose secrets
- methods summaries
- candidate rationale at a non-overstated level
- validation criteria and decision gates
- public-facing summaries and nontechnical briefs
- staged funding logic
- reproducibility notes
- negative-result publication commitment

## Keep Private Or Review Carefully First

These materials should stay private or be reviewed before publication:

- personal phone numbers, email addresses, and direct contact details
- internal sponsor-targeting notes
- private contact lists and outreach routing details
- materials that could overstate experimental status
- unsafe or context-free handling instructions better reserved for qualified labs
- any files containing credentials, account tokens, or environment details

## Good Rule

If a file is mainly for:

- scientific transparency, it is probably public
- sponsor targeting, private outreach, or direct contact operations, it is probably private

## Before Publishing

Review every file for:

- personal information
- overclaiming
- unsupported certainty
- hidden secrets
- accidental release of private notes
